# Phase 2: 写作执行

## 进入条件
- context-brief.md 存在且有章纲
- 作者说「开始写」「继续」「下一章」

## 核心循环

```
┌──────────────────────────────────────────┐
│                                          │
│  [1] 章前确认（narrator + 作者）          │
│       ↓                                  │
│  [2] 正文撰写（narrator）                 │
│       ↓                                  │
│  [3] 快速审查（consistency-checker）      │
│       ↓                                  │
│  [4] 作者决定：修改 / 继续                │
│       ↓                                  │
│  [5] 追踪更新（narrator）                 │
│       ↓                                  │
│  [6] 下一章 ──────────────────────→ 循环  │
│       ↓ 剩余 ≤3 章时触发                    │
│  [7] 章纲续期检查 ← 嵌入核心循环            │
│                                          │
│  每 10 章 → checkpoint 深度审查           │
└──────────────────────────────────────────┘
```

---

## [1] 章前确认

**执行者：** narrator Agent（加载 `references/agents/narrator.md`）

```
narrator 加载：
  - context-brief.md（全局快照）
  - 当前章纲
  - 出场角色的档案（仅本章涉及的）
  - 相关伏笔条目
  - 前一章正文（文风锚点）

narrator 向作者展示：
  「第 N 章准备开始。
   本章目标：{章纲要点}
   情绪基调：{建议}
   新增信息：{N 个}
   出场角色：{列表}
   待回收伏笔：{如有}
   
   这个方向可以吗？有什么要调整的？」
```

## [2] 正文撰写

```
narrator 执行写作

遇到特殊场景时：
  → 询问作者是否调 scene-specialist
  → 作者同意 → 加载 → 给建议 → 融入正文

发现大纲冲突时：
  → 暂停
  → 告知冲突：「大纲说 X，但前面第 N 章发生了 Y」
  → 提出修正方案
  → 等待作者决定

写作完成后展示正文
```

## [3] 快速审查

**执行者：** consistency-checker Agent（加载 `references/agents/consistency.md`）

```
快速检查模式：
  - 范围：本章 + 与前一章的衔接
  - 深度：事实层面为主、逻辑层面为辅
  - 输出精简报告

展示：「快速审查完成。发现 N 个问题：
  🔴 必须修：{数量}
  🟡 建议修：{数量}」
```

## [4] 作者决定

```
「要修改本章还是继续下一章？」
  → 修改 → narrator 修改正文 → 回到步骤 [3] 重新审查
  → 继续 → 进入步骤 [5]
  → 回头调整大纲 → 退回 Phase 1 相关步骤
```

## [5] 追踪更新

```
narrator 更新：
  1. 伏笔账本 — 新增埋设 + 标记回收
  2. 角色状态 — 每个出场角色的当前快照
  3. 章节摘要 — 本章 3-5 句压缩
  4. 时间线 — 追加本章关键事件及时间节点
  5. context-brief.md — 更新进度、最近 5 章、角色速览
  6. 爽点节奏 — 标记本章爽点类型与强度（网文项目）
  7. 会话日志 — 追加本次写作记录
```

## [6] 继续下一章

```
narrator：「第 N 章完成。要写第 N+1 章吗？」

循环回到步骤 [1]
```

---

## [7] 章纲续期检查（每章后自动）

每章追踪更新完成后，narrator 自动检查剩余章纲数量：

- 剩余 > 3 章：静默通过
- 剩余 ≤ 3 章且 > 0：提示「章纲快用完了，要现在生成下一批吗？」
- 剩余 = 0：强制阻断「章纲已用完，必须先让 architect 生成新章纲才能继续」

---

## Checkpoint: 10 章深度审查

每完成 10 章触发：

### 执行者：consistency-checker（深度模式）

```
全局扫描：
  ├── 事实一致性（全篇交叉比对）
  ├── 逻辑一致性（因果链/动机链/规则自洽）
  ├── 结构一致性（节奏/信息密度/篇章比例）
  └── 缺口扫描（大纲缺失/设定空白/伏笔断线）

输出完整审查报告 → 审查/全局审查报告.md + 审查/一致性检查.md → 与作者协商
```

### 审查后协商

```
「前十章全局审查完成。
  🔴 必须修 N 项
  🟡 建议修 M 项
  
  另外，后面的大纲需要调整吗？前十章写下来有没有发现方向需要微调的地方？」
```

---

## 中断与恢复

- 作者任何时刻说「停」「先不改」「先不写了」→ 保存当前状态，更新 context-brief.md
- 下次说「继续写」→ 从 context-brief.md 恢复，从上次中断的章节继续

## 特别注意

- 写作中可以回头调整大纲，但必须和 architect 协商
- 每章写完后必须更新追踪文件，不得跳过
- Checkpoint 时如果发现大量问题（>10 个），建议暂停写作先修复
- 快写优先——不要因为纠结一句话而卡住。可以标记「此处待润色」，精修阶段处理

## 章纲归零兜底

当章纲数量归零（写完全部章纲且未续期），narrator 在步骤 [6] 后强制触发：

```
narrator：「章纲已用完，必须生成新章纲才能继续。要我叫 architect 吗？」
  → 作者同意 → 加载 architect Agent → 生成新章纲 → 确认后继续
  → 作者拒绝 → 等待作者指令
```
