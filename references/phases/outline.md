# Phase 1: 大纲构思

## 进入条件
- 无 context-brief.md 或用户要求「重新构思」

## 流程

### Step 1: 想法捕获与确认

**执行者：** architect Agent（加载 `references/agents/architect.md`）

```
用户输入想法
    ↓
architect 回复：
  - 复述理解：用 1-2 句概括你认为作者想写什么
  - 如想法较模糊 → 提问：最想探索的方向/最想写的场景/最想塑造的角色
  - 如想法较完整 → 直接进入 Step 2
```

### Step 2: 方向建议与协商

```
architect 基于类型 reference 和用户想法，提出 2-3 个故事方向
  - 每个方向：核心概念 + 主角定位 + 差异化
  - 给出推荐：「我倾向 X 方向，因为…你觉得呢？」
    ↓
作者回应 → 协商调整 → 明确方向
```

### Step 3: 结构设计

```
architect 输出三幕/卷级结构
  - 每幕/卷的核心事件和转折点
  - 高潮设计和结局走向

确认方式：先展示整体结构，再逐个节点讨论
「这个整体框架你感觉对吗？我们一个个节点过一下？」
```

### Step 4: 冲突线设计

```
architect 展示核心冲突和多条次级冲突
  - 每条冲突：谁 vs 谁/什么、为什么、如何升级
  - 确认冲突之间的交织关系

「这些冲突线的密度和强度你感觉够吗？有没有想增减的？」
```

### Step 5: 角色设计

**执行者：** character-designer Agent（加载 `references/agents/character.md`）

```
基于大纲设计核心角色
    ↓
逐个展示角色卡 → 协商调整
  - 重点确认：说话方式字段、行为边界
  - 每 1-2 个角色暂停一次

角色关系图 → 协商确认
  - 确认核心关系对的张力和情感底色
```

### Step 6: 章纲生成

```
architect 产出首批章纲（10 章）
  - 每章：目标 + 事件 + 信息增量 + 情绪 + 钩子

确认方式：每 3-5 章展示一次
「前 5 章的节奏和方向你感觉怎么样？要不要调整哪章？」
```

### Step 7: 项目模板部署

```
复制模板文件到用户项目目录：
  - 网文模式 → templates/project-web/
  - 传统文学模式 → templates/project-trad/

替换模板中的占位符：
  {书名} → 作者确认的书名
  {日期} → 当前日期

填充已生成的内容到对应模板文件：
  - 将故事大纲写入 大纲/故事大纲.md
  - 将角色卡写入 角色/ 目录
  - 填充 读者画像.md（网文）/ 世界观文件 / 主题线索 / 人物弧线全景

生成 context-brief.md（基于当前大纲总结）
```

### Step 8: 阶段完成

```
告知作者：
  「大纲阶段完成。你的项目在 {路径}/
   接下来进入写作阶段。要开始写第一章吗？」

等待作者确认 → 转入 Phase 2
```

## 协商密度建议

| 步骤 | 协商频率 |
|------|---------|
| 方向设计 | 高（方向决定一切，反复协商） |
| 结构设计 | 中（确认整体框架，细节可后调） |
| 角色设计 | 高（每个核心角色都要作者点头） |
| 章纲生成 | 中（确认节奏方向，细节写作时可调） |

## 特别注意

- 如果作者中途说「方向不太对」，退回 Step 2 重新来过
- 如果角色设计时发现与大纲冲突，优先修改大纲
- 章纲不是镣铐——写作阶段允许合理调整
