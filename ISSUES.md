## 一、核心矛盾：长章节 vs 评论覆盖率

### 问题描述

工具验证需要两个条件同时满足：
1. **完整章节**（3000+ 汉字） — 短文本（1000 字）不足以触发 style_lint、emotion 等检测
2. **豆瓣评论** — 用于作为 ground truth 对比验证

但当前所有免费源满足其一不满足其二：

| 来源 | 章节长度 | 豆瓣评论 | 可用书数 |
|------|---------|---------|---------|
| bqglll.cc（笔趣阁系列） | ~1000 字预览 | ✅ 31 条 | 13 本 |
| trxs.cc（同人小说） | 2300-4400 字 | ❌ 2 条 | 7 本 |
| biquhe.com 等 9 站 | ~1000 字 | — | 不可用 |
| xbiquge.com | 被 Cloudflare 封锁 | — | 不可用 |

### 影响

- 使用 bqglll.cc 短章节验证：工具 matched 率 61.6%，剩余 37 gaps 均为短文本固有局限
- 使用 trxs.cc 长章节验证：工具 100% matched，但仅 1/7 本书有评论（同人小说豆瓣无条目）
- **无法获得既长又有评论的验证数据集**

### 调研方向

1. 寻找同时提供完整内容和豆瓣条目的免费源
2. 用 playwright 抓取 bqglll.cc 非预览页面（绕过 `?get=content` 限制）
3. 寻找替代评论源（贴吧、知乎、起点本章说）
4. 使用预构建的中文小说语料库 + 豆瓣评论数据集

## 二、Playwright WSL 环境限制

### 问题描述

playwright chromium 需要大量系统依赖（libnspr4、libnss3 等 121 个缺失依赖），WSL 环境下无 sudo 权限无法通过包管理器安装。

### 解决方案

- 通过 `apt download` + `dpkg-deb -x` 手动提取 3 个关键 .so 文件
- 通过 `LD_LIBRARY_PATH` 注入 conda lib 目录
- chromedriver headless shell 可正常启动

### 当前状态

✅ 已解决，playwright 可正常使用

## 三、盗版站统一预览限制

### 问题描述

所有 biquge 系列盗版站（15+ 个域名测试过）统一返回 ~1000 字的片段预览，这是服务端代码逻辑，无法通过客户端手段（cloudscraper、playwright、不同 UA）绕过。

完整章节仅存在于：
- 正版平台（起点/纵横等，需付费 + 登录 + 反爬严格）
- JS 渲染站点（trxs.cc 等，需 playwright 但书单也是 JS 渲染的）

### 已尝试方案

| 方案 | 结果 |
|------|------|
| cloudscraper + 不同参数 | 始终 1000 字 |
| playwright 渲染 | 始终 1000 字 |
| 9 个镜像站切换 | 全部相同限制 |
| Referer 模拟正常浏览 | 无效 |
| 不同 cookie/UA | 无效 |

### 未尝试方案

- 通过起点/纵横 App API（需逆向工程）
- 付费购买章节 + OAuth 登录（成本高）

## 四、评论覆盖率不足

### 问题描述

13 本 bqglll.cc 书中仅 5 本（38%）匹配到豆瓣评论。主要原因：

1. 豆瓣搜索对网文书名匹配率低（尤其是附带有"完结""什么软件能看"等后缀的书名）
2. 部分网文豆瓣没有条目
3. 代理连接不稳定

### 已实施改进

- 书名清洗函数 `_clean_book_title()` 去除噪音后缀
- 修复豆瓣搜索结果的 URL 编码提取
- 添加否定检测（"少了老套"不触发负面）
- 贴吧抓取代码已就绪但未启用（缺少代理环境下的稳定连接）

### 建议改进

1. 启用贴吧 + 知乎作为补充评论源
2. 使用豆瓣 API（而非页面抓取）提高稳定性
3. 增加手动评论种子注入功能

## 五、style_lint 短文本检测能力

### 问题描述

style_lint 的规则引擎基于 cheatsheet_zh.json（50+ 条规则），在 1000 字文本上几乎无法触发（summary.total = 0）。

### 已实施改进

- 新增 `quick_scan()` 函数：副词密度、句首重复、感叹号密度、叠词检测
- 自适应阈值：<2000 字降为 0
- 全书级汇总 `style_lint_book`

### 效果

- quick_scan 在 1000 字文本上有时能发现 1-2 个 issue
- 但仍不足以匹配读者评论的"啰嗦""废话"关键词

### 建议

- 需要 2000+ 字的完整章节才能让规则引擎有效工作
- 在短文本上可以改用统计启发式（n-gram 熵、句子长度分布等）

## 六、emotion 方差归一化

### 问题描述

短文本（1000 字）窗口少（3-5 个滑动窗口），方差虚高。读者说"平淡"但 `intensity_variance` 在 0.09-0.19 之间（高于阈值 0.08）。

### 已实施改进

- 短文本方差乘惩罚系数（chars/2000）
- 从 `avg_intensity` 迁移到 `intensity_variance`

### 效果

- 归一化后方差降低 40-50%
- 剩余 13 个 emotion gap 的 variance 在 0.089-0.102 之间（仅略高于阈值）
